# Kill Switch — параноидальная модель

Защита от любой утечки трафика мимо VPN. Работает с любым VPN-клиентом, который создаёт стандартный VPN-интерфейс типа `ras`.

## Главное правило

> Когда firewall kill switch активен — **весь сетевой трафик** идёт **только через VPN-интерфейс**. В обход не идёт ничего, никогда. Единственные исключения — узкий whitelist: сам VPN-сервер по IP:PORT (чтобы VPN мог дозвониться), loopback, DHCP, опционально локалка и опционально входящие порты.

## Два режима

### Firewall (always-on, нужен админ)

Активируется при запуске программы. **Правила остаются в Windows Firewall навсегда** — даже когда программа выключена, реблудится комп, или прога упала. Интернет полностью заблокирован пока правила есть.

### Tunnels_only (без админа)

Не настоящий kill switch — реакция на падение VPN: останавливает зависимые туннели + убивает указанные процессы. Для лёгких сценариев.

## Конфиг

```json
{
  "kill_switch": {
    "enabled": true,
    "mode": "firewall",
    "vpn_tunnel": "OpenVPN",
    "kill_processes": [],
    "firewall": {
      "vpn_server_ips": ["185.220.101.42"],
      "vpn_server_ports": [1194],
      "vpn_protocols": ["udp"],
      "allow_local_network": true,
      "cleanup_on_exit": false,
      "inbound_allow": [
        {
          "enabled": true,
          "name": "RDP",
          "port": 3389,
          "protocol": "tcp"
        },
        {
          "enabled": false,
          "name": "SSH",
          "port": 22,
          "protocol": "tcp",
          "remoteip": "192.168.0.0/16"
        }
      ]
    }
  }
}
```

### Только IP, без DNS-имён

В `vpn_server_ips` — **только IPv4-адреса**. DNS-имена не поддерживаются: если их разрешать, нужен DNS-трафик, который сам по себе утечка.

**Как узнать IP VPN-сервера:** в обычной командной строке (до включения kill switch):

```cmd
nslookup vpn.example.com
```

Скопируй IP из ответа в конфиг. Если у провайдера несколько серверов — добавь все:

```json
"vpn_server_ips": ["1.2.3.4", "1.2.3.5", "1.2.3.6"]
```

Если IP меняется со временем — нужно перезапускать программу или обновлять конфиг вручную.

### cleanup_on_exit: false (важно!)

По умолчанию `false` — это параноидальная модель:

- При запуске программы правила добавляются (с очисткой старых)
- Программа работает — правила активны
- Программа закрывается через "Выход" — **правила остаются в firewall**
- Программа упала / комп выключился — **правила тоже остаются**
- Следующий запуск программы — старые правила чистятся, новые добавляются (на случай если IP сервера обновился)

То есть **между запусками программы интернет тоже заблокирован** — кроме того что в whitelist. Это и есть параноидальный always-on kill switch.

Если хочешь чтобы интернет работал когда программа выключена — поставь `cleanup_on_exit: true`. Тогда выход из программы снимает все правила.

### Входящие правила (inbound_allow)

Список разрешённых входящих коннектов в обход VPN. Каждое правило — словарь:

| Поле | Что значит |
|---|---|
| `enabled` | `true`/`false` — включено ли это правило |
| `name` | произвольное имя (для логов) |
| `port` | локальный порт |
| `protocol` | `"tcp"` или `"udp"` |
| `remoteip` | (опц.) откуда разрешать. Можно подсеть: `"192.168.0.0/16"` или один IP |

**Примеры:**

Входящий RDP только из локалки:
```json
{
  "enabled": true,
  "name": "RDP from LAN only",
  "port": 3389,
  "protocol": "tcp",
  "remoteip": "192.168.0.0/16"
}
```

Веб-сервер открыт всему миру:
```json
{
  "enabled": true,
  "name": "Web",
  "port": 443,
  "protocol": "tcp"
}
```

Несколько правил можно перечислить — каждое независимо:
```json
"inbound_allow": [
  {"enabled": true, "name": "RDP", "port": 3389, "protocol": "tcp"},
  {"enabled": true, "name": "SSH", "port": 22, "protocol": "tcp"},
  {"enabled": false, "name": "Game server", "port": 25565, "protocol": "tcp"}
]
```

## Что всегда разрешено (нельзя отключить)

Даже с включённым kill switch **постоянно** разрешено:

- `127.0.0.0/8` (loopback) — для локальных туннелей через 127.0.0.1
- DHCP (UDP 67/68) — иначе после ребута сеть не получит IP
- Указанные VPN-серверы по IP:PORT
- Трафик через VPN-интерфейс (всё, что туда отправляется)

Опционально (через конфиг):
- Локальная подсеть (`allow_local_network`)
- Входящие правила (`inbound_allow`)

## Запуск с правами админа

Firewall режим требует админ-прав. Способы:

**Однократно:** ПКМ на .exe → "Запуск от имени администратора".

**Постоянно:** ПКМ на .exe → Свойства → Совместимость → "Запускать от имени администратора". Двойной клик теперь всегда от админа.

**Автостарт + админ:** реестровый автостарт (`HKCU\...\Run`) не даёт админ-прав. Нужен Task Scheduler с галочкой "Run with highest privileges". Скажи если нужно — добавлю автогенерацию задачи.

## Если интернет пропал и нужно срочно восстановить

Открой **обычный cmd от админа** (не из программы, а из меню Пуск → ПКМ "Командная строка" → Запуск от админа) и выполни:

```cmd
netsh advfirewall firewall delete rule name=TunnelTrayKillSwitch_BlockAllOut
```

Это снимет только правило блокировки. Allow-правила безвредны (они без блокировки ничего не блокируют).

Полная чистка через PowerShell от админа:

```powershell
Get-NetFirewallRule -DisplayName "TunnelTrayKillSwitch*" | Remove-NetFirewallRule
```

## Типичные ошибки

**VPN не подключается.** Проверь `logs/kill_switch.log` — там должно быть `[FW] VPN-серверы: [...]`. Если список пустой — IP не прошли валидацию (возможно указано DNS-имя). Если есть — проверь что `vpn_server_ports` и `vpn_protocols` правильные для твоего VPN.

**Интернет заблокирован, программа не запускается.** Открой cmd от админа и выполни команду выше для удаления `BlockAllOut`. Программа умеет восстанавливать правила при запуске — может она не успевает стартовать из-за проблем с PyInstaller-распаковкой? Логи смотри.

**Локалка не работает.** Поставь `"allow_local_network": true`.

**Не могу зайти по RDP/SSH в свою машину снаружи.** Добавь правило в `inbound_allow` с нужным портом.

## Жизненный цикл

1. **Установка:** скачал .exe, положил рядом конфиг с правильными IP VPN-серверов
2. **Первый запуск (от админа):** программа добавляет правила в firewall, интернет блокируется кроме VPN. VPN-туннель из конфига поднимается. Зависимые туннели через `depends_on` стартуют после готовности VPN
3. **Работа:** всё идёт через VPN. Если VPN падает — переподключается через свой whitelist по IP. Туннели через `depends_on` останавливаются и поднимаются обратно
4. **Выход из программы:** правила в firewall **остаются**. Интернет по-прежнему заблокирован
5. **Перезагрузка ПК:** правила всё ещё в firewall. До запуска программы интернета нет совсем (VPN тоже не работает — некому установить туннель)
6. **Следующий запуск:** программа чистит правила и добавляет заново (на случай обновления IP). Цикл повторяется

## Важное предупреждение

Если ты используешь `cleanup_on_exit: false` (по умолчанию) — **никогда не удаляй .exe программы без отключения kill switch**. Иначе у тебя на машине останутся firewall-правила, а программы, которая их умеет убирать, не будет.

Перед удалением: либо измени `cleanup_on_exit: true` и выйди через меню, либо вручную убери правила через `netsh`/PowerShell как описано выше.
