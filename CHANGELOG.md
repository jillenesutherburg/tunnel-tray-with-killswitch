# Changelog

## v4 — Надёжный kill switch для OpenVPN GUI

### 🔧 Переписан движок firewall (главное)

- Движок переведён с `netsh` на PowerShell NetSecurity (`New-/Remove-NetFirewallRule`, `Set-NetFirewallProfile`). Это даёт два возможности, недоступные в `netsh advfirewall`: разрешение по **процессу** (`-Program`) и по **имени адаптера** (`-InterfaceAlias`).
- **Разрешается процесс `openvpn.exe`** ходить наружу — он сам резолвит DNS и коннектится к серверу (в т.ч. failover). Убирает две частые причины локаута: (1) `.ovpn` с именем сервера → DNS режется → VPN не коннектится; (2) необходимость вручную вести whitelist IP. DNS-утечки нет: наружу мимо туннеля ходит только сам openvpn.exe.
- **Разрешается VPN-интерфейс по имени** (автодетект TAP-Windows/Wintun) вместо угадывания подсети туннеля. Раньше неверная `vpn_tunnel_subnet` (или мёртвое правило `interfacetype=ras`, которое для OpenVPN не работает) приводила к тому, что трафик ВНУТРИ туннеля резался. Подсеть/IP остались как запасной вариант.
- Все правила теперь в firewall-**группе** → cleanup одной командой (`Remove-NetFirewallRule -Group`) вместо ~150 отдельных вызовов `netsh` на каждом старте.

### 🚨 Аварийное снятие kill switch

- При активации рядом с программой создаётся **`ОТКЛЮЧИТЬ-killswitch.bat`** — запуск от админа гарантированно снимает правила и возвращает политику firewall, даже если сама программа не стартует.
- Новый пункт меню **«🚨 Снять Kill Switch сейчас»**.
- Исправлена документация по восстановлению: старая инструкция ссылалась на правило `BlockAllOut`, которого давно нет в коде, и не сбрасывала политику. Теперь: удаление по группе + `netsh advfirewall set allprofiles firewallpolicy blockinbound,allowoutbound`.

### 📡 Надёжнее мониторинг OpenVPN

- Новый тип health-check **`openvpn`**: VPN считается живым только если запущен процесс `openvpn.exe` И (если задан host) проходит пинг внутри туннеля. Раньше голый ping мог давать ложный «healthy».

## v3.2 — Фикс активации firewall + автозапуск от админа без UAC

### 🐛 Критический фикс активации kill switch

- `netsh advfirewall firewall add rule` **не поддерживает** параметр `group=` — Windows возвращал `'group' is not a valid argument for this command`, и первое же правило (`BlockAllOut`) падало, из-за чего kill switch не активировался даже под админом. Параметр `group=` убран из всех правил. Правила по-прежнему именуются с префиксом `TunnelTrayKillSwitch_`, и cleanup находит их по имени — группа не нужна.

### ✨ Автозапуск от администратора без UAC

- Новый пункт меню **«Автозапуск от админа (без UAC)»**. Создаёт задачу в Планировщике Windows (`schtasks /SC ONLOGON /RL HIGHEST`) — программа стартует elevated при входе в систему, без запроса пароля/подтверждения. Точно так же автозапускается OpenVPN GUI.
- Включается один раз при запуске от админа. Состояние кэшируется чтобы не дёргать `schtasks` при каждой перерисовке меню.
- Обычный автостарт и admin-автозапуск взаимоисключающие: включение одного выключает другой (иначе при входе была бы гонка двух запусков, и mutex мог бы оставить неэлевейтнутый инстанс → kill switch бы не активировался).
- Пункт меню «Автостарт Windows» переименован в «Автостарт Windows (обычный)» для ясности.

## v3.1 — Поддержка внешнего VPN-клиента и фикс kill switch для OpenVPN

### 🛡 Критический фикс kill switch для OpenVPN/WireGuard

- Правило `interfacetype=ras` разрешает трафик только через **встроенный** Windows-VPN (L2TP/IKEv2/SSTP/PPTP). TAP/Wintun-адаптер OpenVPN классифицируется как `lan`, а не `ras` — поэтому при активном kill switch трафик **внутри** OpenVPN-туннеля блокировался.
- Добавлен параметр `vpn_tunnel_subnet` (напр. `"10.8.0.0/24"`) в `firewall`-конфиг. Создаёт правило, разрешающее исходящий трафик с локальным IP из подсети туннеля — это и есть трафик приложений, который OpenVPN заворачивает в TAP-адаптер. Зашифрованные пакеты при этом уходят на IP VPN-сервера (тоже в whitelist), всё остальное блокируется.

### ✨ Monitor-only режим (для OpenVPN GUI и других внешних клиентов)

- У туннеля появилось поле `"monitor_only": true`. В этом режиме программа **не запускает** `.bat` — только периодически проверяет health-check и выставляет статус HEALTHY/UNHEALTHY. Нужно когда VPN поднимается внешним клиентом (OpenVPN GUI, официальный WireGuard-клиент и т.д.), а программа должна лишь знать жив ли он, чтобы стартовать зависимые туннели и индицировать статус.
- `monitor_only` туннели не требуют `bat_path` (валидация это учитывает).
- Зависимые туннели (`depends_on`) работают как обычно — стартуют когда monitor-only VPN становится HEALTHY, останавливаются при его падении.

### 🧹 Шаблон конфига

- Дефолтный `openvpn` туннель теперь `monitor_only: true` — типичный сценарий с OpenVPN GUI.
- Добавлен `vpn_tunnel_subnet` в шаблон firewall с пояснением.

## v3.0 — Финальный релиз

### 🛡 Безопасность

- **Защита от случайной блокировки интернета.** Если в `vpn_server_ips` остался плейсхолдер `1.2.3.4` из шаблона конфига, firewall kill switch отказывается включаться с понятной ошибкой в `logs/kill_switch.log`. Раньше — если пользователь включал KS до того как поменял IP, интернет блокировался "в `1.2.3.4`".
- **Дефолтный `vpn_server_ips` — пустой список** вместо плейсхолдера. Чтобы случайно запущенный KS просто не включился, а не блокировал интернет в несуществующий IP.
- **`cleanup_firewall_rules` запускается только если `kill_switch.enabled=true`** в конфиге. Раньше — на каждом старте с админ-правами, что неожиданно стирало правила если пользователь выключил KS в конфиге, но хотел чтобы правила оставались (параноидальная модель).

### 🐛 Багфиксы

- **Single-instance защита** через именованный mutex Win32. Двойной клик по `.exe` или случайный второй запуск из автозагрузки больше не плодят дубликаты иконок в трее, борющиеся за одни и те же ресурсы.
- **Race condition в `enable()`/`disable()`/монитор-потоке.** `disable()` теперь ждёт завершения монитор-потока (до 3 сек). `enable()` дожидается старого потока (до 2 сек) перед запуском нового. Раньше быстрый toggle мог породить два параллельных монитор-потока для одного туннеля.
- **`enable()` и `disable()` идемпотентны.** Повторный вызов — no-op.
- **Утечка `_log_file` при сбое `Popen`.** Файл лога открывался до `subprocess.Popen`, и если Popen падал, дескриптор оставался открытым до GC. Теперь файл присваивается в `self._log_file` только после успешного `Popen`, иначе закрывается в `except`.

### ✨ Новые возможности

- **Валидация конфига на старте.** Проверяется: пустые/невалидные имена, дубликаты имён туннелей, несуществующие `depends_on`, циклы в зависимостях (через DFS), плейсхолдер IP в firewall-конфиге, `vpn_tunnel` указывающий на несуществующий туннель. Все проблемы пишутся в `logs/config_errors.log`. В трее появляется отдельный пункт меню "⚠ Ошибок в конфиге: N (открыть)" и всплывающее уведомление при старте.
- **Программа запускается с битым конфигом** — но только с теми туннелями, которые валидны. Раньше непонятная ошибка падала в никуда (под `--noconsole` пользователь её не видел).
- **Ротация логов туннелей.** Когда `<имя>.log` достигает 5 МБ, при следующей записи оставляются последние 2 МБ + строчка `[log rotated at ...]`. Без рантайм-ротации логи могли расти в гигабайты на флапающих туннелях.
- **Debounce уведомлений в трее.** Одинаковое сообщение не показывается чаще раза в 60 сек. Раньше флапающий туннель мог плодить ban-spam уведомлений каждые 10–15 сек.
- **Backoff расширен** с `[5, 10, 30]` до `[5, 10, 30, 60, 120]` сек. На реальных VPN-падениях по 5+ минут это снижает количество спам-реконнектов.
- **В логах теперь полная дата** (`YYYY-MM-DD HH:MM:SS`), а не только время. Полезно когда программа работает дни/недели подряд.
- **Иконка для `.exe`** — `app.ico` (multi-size, поддерживает 16–256 пикселей). Раньше билд имел дефолтную иконку Python.

### 🧹 Качество

- Дефолтные имена туннелей в шаблоне конфига короткие (`openvpn`, `ssh_db`, `standalone`) вместо длинных вроде "OpenVPN (опционально, можно удалить)". Они идут в имена лог-файлов, в `depends_on` и в логи — раньше выглядели грязно.
- Поле `_name_help` в шаблоне напоминает что `name` — это идентификатор, а не описание.

### ⚠ Известные ограничения

- В `firewall` режиме при отсутствии админ-прав туннели всё равно стартуют, но без защиты firewall. Это намеренно — kill switch и так в `tunnels_only` режиме что-то делает. В логе будет warning.
- `pystray.Icon` мутируется из фонового потока `_periodic_refresh` каждые 2 сек. На большинстве систем работает стабильно. Если будут подвисания меню — см. issue tracker.
